# Qard — бесплатный web-режим + удаление мнемоник

Дата: 2026-06-30
Ветка: `feature/free-tier-web`
Точка возврата: тег `stable/ownkey-v1`

## Цель

Сделать web-версию Qard бесплатной: пользователю не нужен свой ключ ИИ.
Запросы идут через Cloudflare Worker, который держит ключ владельца в секрете
и ограничивает нагрузку. Параллельно — продуктовые лимиты бесплатного тарифа
и удаление фичи мнемоник.

Desktop (Electron) **не меняется**: работает напрямую с ProxyAPI по своему
ключу, без дневного лимита, поле ключа на месте.

## Не-цели (YAGNI)

- Подписки/монетизация — позже, когда появятся спонсоры.
- Авторизация пользователей, серверный учёт по аккаунту.
- Защита от распределённой атаки по многим IP (для MVP приемлемо).

## Архитектура

Две ветки исполнения ИИ:
- **Web:** `src/web/backend/ai.ts` → Cloudflare Worker `qard-ai-proxy` → ProxyAPI.
- **Desktop:** `src/main/services/gemini.service.ts` → ProxyAPI напрямую (без изменений).

Платформа определяется build-флагом `__IS_WEB__` (DefinePlugin): `true` в
`webpack.web.config.js`, `false` в renderer-конфиге `webpack.config.js`.

## Компоненты

### 1. Worker `qard-ai-proxy` (сервер, только web)

Файлы: `worker/src/index.ts`, `worker/wrangler.toml`.

Поведение:
- Секрет `PROXYAPI_KEY` (env binding). Заливается владельцем вручную, в коде
  его нет.
- KV-namespace `RL` (биндинг в `wrangler.toml`) для дневного счётчика.
- Логика на POST:
  1. Если есть заголовок `Authorization: Bearer <key>` (продвинутый режим,
     свой ключ) → форвардим с этим ключом, **лимит не применяем**.
  2. Иначе (бесплатно): ключ = `gen:<CF-Connecting-IP>:<дата-UTC>`.
     - Запись в KV есть → `429 {"error":"daily_limit"}`.
     - Нет → подставляем `env.PROXYAPI_KEY`, форвардим, при успехе пишем
       запись в KV c TTL до конца суток UTC.
  3. Защита от разорения одним запросом: лимит размера тела (например > 8 КБ →
     `413`) и потолок `max_tokens` в исходящем запросе.
- CORS — как сейчас (whitelist origin'ов), без изменений.
- IP: `request.headers.get('CF-Connecting-IP')`.

Замечание о KV: бесплатный тариф ~1000 записей/сутки (= 1000 бесплатных
генераций/сутки на всех). Eventually-consistent — теоретически можно
проскочить 2 запроса в одну секунду; для дневного лимита незначимо.

### 2. Web-клиент ИИ `src/web/backend/ai.ts`

- Ключ больше не обязателен. Если в настройках есть свой ключ (продвинутое) —
  шлём `Authorization: Bearer <key>`; иначе без него.
- Обработка `429` → типизированная ошибка (например `class DailyLimitError`),
  которую UI показывает как «Лимит на сегодня исчерпан».
- Удаляется метод `mnemonic` (фича мнемоник убирается).

### 3. Настройки (Setup) — `Settings.tsx` + `Settings.css`

- **Web (`__IS_WEB__`):**
  - Инфо-блок лимитов сверху секции AI: «Бесплатно: 1 генерация в день,
    до 30 слов. Модель: <текущая>».
  - Поле ключа спрятано в сворачиваемый блок «Продвинутое»; подпись:
    свой ключ снимает дневной лимит.
  - Остаются селекторы: Модель, Количество примеров.
- **Desktop:** без изменений (поле ключа, провайдер, модель на месте).

### 4. Общие продуктовые лимиты (обе платформы)

- Кол-во примеров: опции `[1,2,3]` (было до 5), дефолт 3.
- Цель обучения (reviews/day): опции `[10,15,20]` (было до 100), дефолт **20**
  (было 30).
- Подрезка старых сохранённых значений при загрузке: `exampleCount → ≤3`,
  `dailyGoal → ≤20`. Места дефолтов: `store/index.ts`,
  `src/web/backend/settings.ts`, `src/main/services/settings.service.ts`.

### 5. Generate — `Generation.tsx`

- На web: `canGenerate = words.length > 0` (без требования ключа); пункт
  валидации про ключ скрыт; summary-пункт «API Key» заменён на счётчик
  «Слов: N / 30».
- Мягкий клиентский лимит 30 слов (обрезка/предупреждение).
- Ошибку дневного лимита показываем сообщением в блоке прогресса/ошибок.

### 6. Удаление мнемоник (обе платформы)

Полностью убрать фичу:
- `gemini.service.ts` — метод `generateMnemonic` + промпт `buildMnemonicPrompt`.
- `web/backend/ai.ts` — метод `mnemonic`.
- `shared/ai-prompts.ts` — `buildMnemonicPrompt`.
- IPC-канал `AI_MNEMONIC`, его handler и вызов в preload/global.d.ts.
- UI-кнопка/вызов мнемоники (найти при планировании).
- Связанные i18n-ключи.

## Поток данных (web, бесплатная генерация)

1. Пользователь вводит ≤30 слов → «Start Generation».
2. `web/backend/ai.ts` POST в Worker без `Authorization`.
3. Worker проверяет KV по IP+дата → нет записи → подставляет секрет-ключ →
   ProxyAPI → ответ; пишет KV с TTL.
4. Карточки парсятся и показываются.
5. Повторная генерация в тот же день → Worker `429` → UI «Лимит исчерпан».

## Обработка ошибок

- `429 daily_limit` → понятное сообщение + (опц.) когда сбросится.
- `413`/слишком большое тело → «Слишком много слов, максимум 30».
- Сетевые/прочие → текущее поведение (сообщение об ошибке генерации).

## Ручные шаги владельца (Cloudflare, требуется логин)

1. `cd worker && wrangler secret put PROXYAPI_KEY` — вставить ключ.
2. `wrangler kv namespace create RL` — id вписать в `wrangler.toml`
   (шаблон биндинга будет подготовлен в коде).
3. `npm run worker:deploy`.

## Тестирование

- Worker: юнит-логика лимита (есть запись → 429; нет → пропуск + запись;
  свой ключ → без лимита; большое тело → 413).
- Web `ai.ts`: 429 → `DailyLimitError`; есть свой ключ → уходит `Authorization`.
- Подрезка настроек: старый `dailyGoal=50` → грузится как 20; `exampleCount=5`
  → 3.
- Регресс desktop: генерация со своим ключом работает как раньше; мнемоник
  нигде нет; сборка `tsc --noEmit` чистая.
